# Worked example — `examples/mini-arena/`

A complete, vetted last-man-standing game in **8 code lines**. Every file passes
`dfpy.py check`, the folder passes `validate` and `audit.py`. When building a NEW game
from scratch, **imitate these files** — they are the smallest correct instance of the
core patterns.

> `examples/` lives under `docs/`, **not** `plots/`, so it is never pulled or pushed to a
> plot. To actually try it, copy the folder into `plots/` (e.g. `plots/mini-arena/`) and
> have the human `/df push`. The lobby pad (10, 51, 25) and arena box (25–45, 51, 15–35)
> assume flat ground — adjust the `loc(...)` calls to the real plot.

## The game in one paragraph
Players idle in a lobby. `@start` throws everyone into the arena with a sword; deaths
knock you out; the last fighter standing wins. Wins / kills / rounds persist forever as
`saved` stats shown by `@stats` and on the lobby action bar.

## State model (decide this FIRST for any game)
| Variable | Scope | Why that scope |
|---|---|---|
| `arenaState` (0 idle / 1 live / 2 winner pending) | `unsaved` | one shared round, wiped on plot stop |
| `alive` | `unsaved` | shared counter for the live round |
| `%uuid in` (0/1) | `unsaved` | per-player runtime flag — `%uuid` keys it per player |
| `%uuid loopGen`, `%uuid hudT` | `unsaved` | loop bookkeeping, worthless after a restart |
| `%uuid wins / kills / rounds` | `saved` | the stats players keep — `%uuid` (never `%default`: names change) |
| `myGen`, `drop` | `local` / `line` | scratch inside one thread / one code line |

## File → pattern map
- **`event__Join.py` — init + welcome.** First-visit `saved` defaults via
  `IfVar('VarExists', …, attribute='NOT')` (never blind-zero a returning player's stats),
  `InstantRespawn` (so the `Respawn` event is the one "after any death" hook), a short
  MiniMessage welcome, `CallFunc('toLobby')`, then `%uuid loopGen += 1` and
  `StartProcess('gameLoop')` — the generation bump kills any stale loop from a rejoin.
- **`process__gameLoop.py` — the per-player loop dispatcher** (`performance.md`). One
  `Repeat('Forever')` with a `Control('Wait')` (no wait = LagSlayer), a stale-generation
  guard (`myGen` is `local`: it belongs to this thread), cheap work every 5 ticks, and a
  `%uuid hudT` counter so the action-bar HUD runs once a second. In a bigger game each
  cadence branch becomes one `CallFunc` per feature. It also claims the win: when
  `arenaState` is 2, only the player still flagged `in` enters the branch — per-player
  loops let "the winner" target themselves with no conditional select.
- **`event__Command.py` — the `@` command router** (`commands.md`). `CancelEvent` first
  (don't echo the raw `@…`), then one `IfGame('CmdArgEquals', …, num('1'))` per command.
  Plot commands are `@start`, never `/start`. Tiny actions inline; real logic routes to a func.
- **`func__startRound.py` — guarded state transition + fan-out.** Re-entry guard →
  `Control('Return')`. `SelectObj('AllPlayers')` + `gval('Selection Size')` counts the
  fighters; broadcast actions take `target='Selection'`, the "need 2 players" error takes
  `target='Default'` (just the sender). Then the canonical per-player fan-out:
  `StartProcess('arenaIn', tags={'Target Mode': 'For each in selection', 'Local Variables': "Don't copy"})`
  — needed because a plain `SetVar` on a selection resolves `%uuid` only once; each player
  must run their own thread to write their own `%uuid in`.
- **`process__arenaIn.py` — one fighter's entry.** Flag `%uuid in`, count the `saved`
  round, kit (item generated by `dfpy mkitem`, never hand-written SNBT), `SurvivalMode`,
  `RandomLoc` + `Teleport` to scatter spawns. `drop` stays `'line'` — it never crosses a
  CallFunc.
- **`event__KillPlayer.py` — score + saved stats.** `SelectObj('EventTarget', tags={'Event
  Target': 'Killer'})` makes the killer the selection explicitly (don't rely on the event's
  default target), so `%uuid kills` credits the right player; `%victim` names the victim.
- **`event__Respawn.py` — the respawn flow.** `Death` only fires for non-player/entity
  deaths, but `Respawn` fires after *every* death (and `InstantRespawn` makes it immediate),
  so the round math lives here: un-flag, `alive -= 1`, and when one fighter remains set
  `arenaState = 2` — the *winner's own loop* does the awarding. Ends in `toLobby`.
- **`func__toLobby.py` — the single reset path.** Every road back (join, death, win,
  `@lobby`) goes through one func: un-flag, heal, clear, adventure mode, teleport. One
  reset func = no half-reset states.

## Why each gotcha-rule appears where it does
- `func`/`process` headers all carry `tags={'Is Hidden': 'False'}`; every `StartProcess`
  passes `Target Mode` + `Local Variables` — omitting either halts the thread in-game
  ("invalid chest parameters") yet passes `check` (`quirks.md`).
- Every action, tag, and option name was grepped in `actiondump.json` before use
  (e.g. the sound is `Pling`, not "Note Block Pling"; the option is `"Don't copy"`).
- `num()` always takes a string; all text is `comp()` MiniMessage — no `§`/`&` anywhere.
- Scopes follow the table above; nothing here crosses a `CallFunc`, so `'line'`/`'local'`
  scratch is safe. The moment a var is shared across files it became `unsaved`/`saved`.
- Longest line is 34 blocks — well under the cap. Keep lines short; split into funcs.

## Build order (how it was actually layered)
1. **Skeleton first:** `event__Join` (stats init + welcome) + `func__toLobby` +
   `event__Command` with only `@stats`. `check` each file → human `/df push` → join, read
   the messages. *Smallest playable thing, verified in-game.*
2. **The loop:** `process__gameLoop` with just the guard + lobby action bar. Push, watch it.
3. **The round:** `func__startRound` + `process__arenaIn`; `@start` should teleport and kit
   everyone (state never ends yet — that's fine, test it anyway).
4. **The ending:** `event__Respawn` + the win-claim branch in `gameLoop`. Now it's a game.
5. **The garnish:** `event__KillPlayer` credits, sounds, titles, `@lobby`.

Each step ends with `check`/`validate` green and a human push-and-test. Never build all
8 files before the first push.

## Checklist for starting any new game
1. Write the **state model table** first: every variable, its scope, why.
2. Sketch the file list: one event per trigger, one func per verb, one loop process.
3. Build in the order above: Join + one command → push → test → layer.
4. For every action/tag: **grep `actiondump.json`**; for every system: grep
   `action-reference.md` before hand-rolling (cooldowns, JSON, … are native).
5. Items via `dfpy mkitem`, text via `comp()` MiniMessage, plot commands via `@`.
6. Every loop has a `Wait`; every loop process has a stale-generation guard.
7. Headers: `Is Hidden` on func/process; both tags on every `StartProcess`.
8. After every edit: `dfpy.py check <files>`, then `validate <folder>` before a push —
   and remember green ≠ fun: apply `game-design.md` and `styling.md` before calling it done.
